# Design: User Skill Library and Cross-Agent Sync

Generated by /office-hours on 2026-05-22
Branch: development
Repo: accordagents
Status: DRAFT
Mode: Builder

## Problem Statement

The app should let a user see the skills that are available to local agent runtimes for the current repo, especially Claude Code and Codex, and make a skill available to the other runtime with a clear sync action.

The user clarified the key product boundary:

> the skills existing in the app - they are internal, for agents to make requsts to other participants, access context, so "Managing skills in the app" feature should not include those

The user also clarified what "sync" means:

> if I see the skill for the repo for claude, I want to press button and make it available for codex

So this feature is not an editor for AccordAgents' hidden app bridge skills. It is a user-facing skill library for repo/global agent skills, with sync between provider-specific runtimes.

## What Makes This Cool

The useful moment is simple: a user notices "Claude has this repo skill, Codex does not," clicks one button, reviews the generated target, and Codex can now use the same workflow.

That turns skill portability from a file-system chore into an explicit product surface.

## Scope

In scope:

- Inventory user-visible skills for the selected repo and local user account.
- Show whether each skill is visible to Claude Code, Codex, both, or neither.
- Automatically detect existing provider-specific skills and represent them in the library.
- Make a library or detected skill available to Claude Code or Codex.
- Show collisions, stale generated copies, validation errors, and provider-specific limitations.
- Hide all app-owned internal bridge skills from normal skill management.

Out of scope for the first version:

- Managing AccordAgents' internal participant-request/context-access skills.
- Editing app MCP tools, app tool capabilities, participant permissions, or chat role prompts.
- Running skill scripts as part of validation.
- Syncing arbitrary Claude plugins or Codex plugins.
- Cloud workspace administration for ChatGPT or Enterprise skill catalogs.
- Manual skill-authoring UI as the primary creation path. Future creation should be participant-driven through app MCP proposals, with user approval.

Future scope:

- Chat participants can propose user-facing skills through app MCP tools.
- The app reviews the proposed skill, shows a preview, and asks the user to approve before writing.
- Approved participant-created skills are rendered to all available supported local runtimes by default, with provider-specific skips shown only when a runtime is missing, unsupported, or unsafe to write.
- Approved participant-created skills enter the same detected/synced skill surface as filesystem-created skills after the app writes the approved provider targets.
- App-created or app-modified skills use the same provider-neutral internal model and render through per-CLI adapters for all supported local runtimes.
- Participant-created skills must still be unable to create or expose internal app bridge skills.

## Existing Context

The repo already has an `AppSkillsService` at `src/main/services/appSkills.ts`. It syncs bundled app-owned skills from `src/main/appSkills` into provider skill folders and marks generated folders with `.accordagents-generated.json`.

That service is infrastructure. It should not become the user-facing skill library because it has the wrong ownership model:

- It treats `src/main/appSkills` as the source of truth.
- It generates names prefixed with `accordagents-`.
- It writes internal skills for app protocol use.
- It intentionally owns and cleans generated targets through manifest and marker files.

The new feature should use some parsing/rendering ideas from `AppSkillsService`, but it needs a separate `UserSkillLibraryService` with separate markers, separate IPC, and separate UI language.

External format notes:

- Claude Code documents personal skills in `~/.claude/skills/` and project skills in `.claude/skills/`, each as a directory with `SKILL.md` plus optional supporting files. The required frontmatter fields are `name` and `description`. Source: [Claude Agent Skills docs](https://docs.claude.com/en/docs/claude-code/skills).
- OpenAI says Skills are supported in Codex, do not sync across products yet, and follow the Agent Skills open standard so they can be downloaded from one product and installed in another. Source: [OpenAI Skills in ChatGPT help article](https://help.openai.com/en/articles/20001066-skills-in-chatgpt).

## Premises

1. Internal app skills are hidden infrastructure, not user-managed content.
2. The user mental model is "make this skill available to Claude/Codex," not "edit manifests."
3. The service should use a provider-neutral internal model, but v1 should not expose or write a canonical library invisibly.
4. Every write action needs a preview and collision protection because skills can contain instructions and executable helper scripts.
5. Repo skills should be first-class when they already exist or when the user explicitly creates them, because team-shared skills matter more than one user's global setup.

## Approaches Considered

### Approach A: Mirror Repo Skills

Summary: Scan provider folders like `.claude/skills` and `.codex/skills`, then copy/translate `SKILL.md` between them on demand.

Effort: M
Risk: Med

Pros:

- Ships the fastest useful version.
- Directly matches "Claude has it, make Codex have it."
- Small data model and fewer migrations.

Cons:

- Provider folders become the source of truth, which makes drift harder to reason about.
- Editing and version history become awkward later.
- Harder to support global, repo, and future provider scopes cleanly.

Reuses: `parseSkillFrontmatter`, safe path handling, and generated marker concepts from `AppSkillsService`.

### Approach B: Full Skill Library

Summary: Add a canonical user-facing skill library in the app, automatically detect existing provider skills, then generate provider-specific outputs for Claude Code and Codex when the user asks to make a skill available somewhere.

Effort: L
Risk: Med

Pros:

- Gives the app a clear source of truth for user-facing skills.
- Makes sync, drift detection, validation, and future editing coherent.
- Can support repo skills, user-global skills, and future providers without redesigning the model.

Cons:

- More UI and service surface before the first sync button works.
- Needs careful language so users never have to understand internal library adoption.
- Requires strict filtering so internal app skills never leak into the library.

Reuses: existing app-owned sync safety patterns, but not the app-owned source root or naming.

### Approach C: Diagnostics First

Summary: Show where Claude and Codex are looking for skills and explain why a skill is missing, but do not write files yet.

Effort: S
Risk: Low

Pros:

- Safest first release.
- Helps debug local setup without mutating anything.
- Useful even when one runtime is not installed.

Cons:

- Does not deliver the user's one-click sync goal.
- Still leaves the user to copy files manually.
- Risks feeling like an inspector rather than a management feature.

Reuses: CLI detection and health display in the current settings screen.

## Recommended Approach

Choose a repair-first v1 with a provider-neutral core:

1. Inventory existing user-visible provider skills automatically.
2. Normalize detected skills into an internal provider-neutral model for validation, comparison, and future authoring.
3. When a user clicks "Make available to Claude" or "Make available to Codex," preview the exact target write, validate it, and write only the selected provider target with ownership markers.
4. Add explicit app-owned skill creation, editing, history, and repo storage later.

This honors the user's product goal while avoiding both a brittle copy-only design and invisible repo writes.

The user should never see an "Import" step. Existing skills appear automatically. The only visible sync action is making a discovered skill available to another runtime.

Important nuance: "repair-first v1" is product framing, not a rejection of a provider-neutral core. The implementation should still normalize detected skills into an internal skill model and render them through provider adapters. That lets a later create/modify feature apply one approved skill to every available CLI without redesigning the service. What should be deferred is exposing that internal model as a user-facing "canonical library" before creation/editing exists.

## Core UX

Add a "Skills" section in settings or the project sidebar.

The primary table:

| Column | Meaning |
| --- | --- |
| Skill | Name, description, and source scope |
| Claude | Available, missing, stale, collision, or invalid |
| Codex | Available, missing, stale, collision, or invalid |
| Source | Repo skill, Claude-only, Codex-only, both, or global |
| Action | Make available to Claude, Make available to Codex, Compare, Validate |

Common flows:

1. User opens Skills.
2. App scans repo and user-global skill folders.
3. Internal app skills are filtered out before display.
4. User sees `release-checklist` available for Claude but missing for Codex.
5. User clicks "Make available to Codex."
6. App shows a diff preview of the exact Codex target files it will write.
7. User confirms.
8. App writes the Codex target and marks it as app-synced user content.

Details drawer:

- Frontmatter summary.
- Skill body preview.
- Supporting files list.
- Provider visibility.
- Last synced hash per provider.
- Validation warnings.
- Generated diff preview before writes.

## Skill Sources and Targets

Explicit repo user skill source (pre-existing or future app-authored; not created by hidden v1 adoption):

```text
.accordagents/skills/<skill-id>/
  SKILL.md
  assets/
  scripts/
  templates/
  .accordagents-skill.json
```

Claude project target:

```text
.claude/skills/<skill-id>/
  SKILL.md
  ...
```

Codex target:

```text
.codex/skills/<skill-id>/
  SKILL.md
  agents/openai.yaml
  ...
```

If Codex repo-local skill loading is not available in the detected local Codex version, sync to a global target with a repo-scoped generated name:

```text
~/.codex/skills/<repo-slug>-<skill-id>/
```

The UI should say which target is active. Do not silently switch between repo and global scope.

## Filtering Rules

Never show these in the user skill manager:

- `src/main/appSkills/**`
- Generated folders with `.accordagents-generated.json` where `owner` is `accordagents`
- App-owned generated folders named `accordagents-*`
- Any skill whose manifest says it is an app bridge/internal protocol skill

Internal bridge skills must never appear in this skill manager, including search, advanced filters, details drawers, sync previews, and diagnostics reachable from this screen. If internal diagnostics are needed later, they belong in a separate developer diagnostics surface with no sync actions.

Show these:

- `.claude/skills/*/SKILL.md`
- `.codex/skills/*/SKILL.md` if supported by local Codex
- `~/.claude/skills/*/SKILL.md`
- `~/.codex/skills/*/SKILL.md`
- `.accordagents/skills/*/SKILL.md` when the folder already exists or when future app-authoring creates it explicitly

When a global folder is app-owned internal infrastructure, hide it. When a global folder is user-authored or app-synced user content, show it. V1 must not create `.accordagents/skills` as a hidden adoption step.

## Data Model

Add shared types in `src/shared/types.ts`:

```ts
export type SkillProvider = "claude-code" | "codex-cli";
export type SkillScope = "repo" | "global";
export type SkillSourceKind = "repo-user" | "provider" | "generated-provider";
export type SkillVisibilityStatus = "available" | "missing" | "stale" | "collision" | "invalid" | "unsupported";

export interface UserSkillDescriptor {
  id: string;
  name: string;
  description: string;
  sourceKind: SkillSourceKind;
  sourceScope: SkillScope;
  sourcePath: string;
  providers: Record<SkillProvider, UserSkillProviderState>;
  warnings: string[];
}

export interface UserSkillProviderState {
  status: SkillVisibilityStatus;
  scope?: SkillScope;
  path?: string;
  sourceHash?: string;
  renderHash?: string;
  message?: string;
}
```

Markers for app-synced user skills should not reuse `.accordagents-generated.json`, because that file currently means internal app skill ownership. Use a new marker:

```text
.accordagents-user-skill.json
```

Marker fields:

- `owner: "accordagents-user-skill"`
- `schemaVersion`
- `skillId`
- `sourcePath`
- `provider`
- `scope`
- `generatedFiles`
- `sourceHash`
- `renderHash`
- `updatedAt`

## Service Design

Add `src/main/services/userSkills.ts`.

Responsibilities:

- Scan user skill locations, including provider folders and explicit `.accordagents/skills` repo skills when present.
- Parse `SKILL.md` frontmatter.
- Filter internal app-owned skills.
- Compare provider outputs against recorded source hashes and render metadata.
- Generate provider-specific render output.
- Build write previews.
- Apply writes atomically with collision checks.
- Return structured health and warning data to the renderer.

Do not let renderer code read arbitrary file paths. All path traversal and symlink checks live in the main process.

IPC surface:

```ts
skills:list(repoPath: string): Promise<UserSkillDescriptor[]>
skills:preview-sync(request: SyncUserSkillRequest): Promise<UserSkillSyncPreview>
skills:sync(request: SyncUserSkillRequest): Promise<UserSkillSyncResult>
skills:validate(request: ValidateUserSkillRequest): Promise<UserSkillValidationResult>
```

## Safety Rules

- Never overwrite an unmarked provider target.
- Never follow symlinks when scanning or writing skill folders.
- Never write outside the selected repo or known user skill roots.
- Always validate frontmatter before syncing.
- Preserve supporting files only if they are normal files under the skill directory.
- Show a write preview before syncing.
- Include a clear warning when syncing a skill with scripts.
- Keep app-owned internal skills out of default UI and out of sync actions.

## Validation

Validation should check:

- `SKILL.md` exists.
- Frontmatter has `name` and `description`.
- `name` is safe for provider targets.
- `description` is non-empty and under provider limits.
- Supporting file paths are safe.
- No symlinks inside the skill directory.
- Codex render can generate `agents/openai.yaml`.
- Claude render preserves Claude-only frontmatter such as `allowed-tools`, but Codex sync warns that Claude-only fields may not carry over.

- Should generated provider outputs be committed, ignored, or explicitly chosen per project?

- A collision never overwrites user-authored provider content.
- A stale generated provider copy is detected after the recorded source skill changes.
- `make typecheck` and `npm run test:app-skills` remain green after the shared parser/refactor.

### Resolved User Challenge

**Challenge 1: Reframe v1 from "Full Skill Library" to "Skill Availability Repair"** (from CEO/Design/Eng/DX)

Original direction: choose Full Skill Library.

User decision: adopt Repair V1.

Review recommends: keep the long-term library direction and provider-neutral service core, but make the first implementation provider-to-provider availability repair: auto-detect existing skills, show which agent is missing a skill, preview the target write, and sync safely. Do not write `.accordagents/skills` invisibly in v1.

Why: every phase found that visible "library/canonical/adopt" language fights the user's mental model. The user wants "make this skill available to Codex." Hidden canonical adoption also risks dirtying the repo unexpectedly.

What we might be missing: you may already know that future skill editing/history/team sharing is central enough to justify a canonical store immediately.

Choose "Keep Full Library" for v1 only if one of these is true:

- Skill creation or editing is part of the same first release.
- Skills must immediately become reviewed/versioned repo assets owned by AccordAgents.
- Team sharing, history, or policy controls are core v1 workflows.
- The app needs to manage skills before any Claude/Codex filesystem source exists.

If we're wrong, the cost is: v1 may be too narrow and require a second migration when full authoring arrives. If the provider-neutral core is skipped, future create/modify across all CLIs becomes a rewrite instead of an extension.

Implementation should therefore use repair-first product language and behavior in v1 while preserving the provider-neutral core needed for future create/modify workflows across all supported CLIs.
