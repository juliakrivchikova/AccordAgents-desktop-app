# Design: Participant Behavior Rules MVP

Generated by /office-hours on 2026-06-02

## Problem Statement

AccordAgents users need a way to create reusable rules for participant behavior without duplicating whole role prompts. A role defines the participant's job, such as Software Engineer or QA Lead. A behavior rule defines reusable operating preferences, such as "be concise", "challenge assumptions", "ask before editing", or "always include verification".

The product goal is:

`role` chooses what kind of specialist the participant is.
`behavior rule` chooses how that participant should behave.

## What Makes This Useful

The useful version is a small rule library that can be attached to saved participants. A user can create one rule like "Keep replies short unless I ask for detail", attach it to several participants, and update it once later.

This prevents prompt duplication and makes participant behavior easier to inspect before starting a chat.

## Constraints

- Behavior rules must not be confused with shell permission rules. Permission rules control command execution. Behavior rules control response style and decision posture.
- Roles remain the primary reusable instruction template. Behavior rules are overlays, not replacements.
- Saved participant presets are the MVP attachment point. Editing saved presets should not rewrite old chat transcripts.
- Existing active chat sessions use role and participant snapshots. Behavior rules must participate in the same snapshot model so behavior changes are coherent across turns.
- A rule edit should affect future participant runs that resolve that saved participant, but old messages remain unchanged.
- Rule deletion must not leave saved participants in a broken state.
- The implementation should fit the existing settings, shared types, IPC/preload, renderer settings, and `ChatService` runtime contract.

## Premises

1. Users will understand behavior rules best as reusable instruction snippets, not as another role type.
2. The first useful scope is saved participants, not global chat overrides or conditional runtime policies.
3. Behavior rules should be resolved by the main process at run time, because the renderer should not own prompt composition.
4. Existing chat sessions should receive a refreshed instruction snapshot when selected rule content changes.
5. Permission controls stay separate because combining behavior and capability rules would make safety harder to reason about.

## Approaches Considered

### Approach A: Put Rules Inside Role Instructions

Summary: Tell users to edit role prompts directly and add their desired behavior instructions there.

Effort: S
Risk: Medium

Pros:
- No new data model.
- Works immediately with existing role editor and runtime session snapshots.

Cons:
- Forces users to duplicate the same behavior across many roles.
- Makes "Software Engineer" and "Software Engineer but concise" into separate roles.
- Hard to inspect which reusable behavior preferences are active on a participant.

Reuses: Existing role editor, role versioning, role snapshot behavior.

### Approach B: Participant Rules MVP

Summary: Add a reusable behavior-rule library in Settings, attach rule IDs to saved participant presets, and merge selected rule instructions into the resolved participant runtime instructions.

Effort: M
Risk: Low

Pros:
- Matches the current role/participant/session model.
- Lets one rule be reused across many saved participants.
- Keeps behavior preferences separate from permissions and role identity.
- Small enough to ship before conditional per-chat rule policies.

Cons:
- Does not yet support per-chat temporary overrides.
- Does not yet support conditions such as "only apply in code review".
- Rule ordering is simple and follows settings order.

Reuses: `AppSettings`, `SettingsService`, `ChatParticipantConfig`, participant draft UI, IPC/preload settings routes, `ChatParticipantSession`, and `ChatService` role resolution.

### Approach C: Full Rule Engine

Summary: Add global rules, participant rules, per-chat overrides, rule priority, enable/disable conditions, and conflict handling.

Effort: L
Risk: Medium

Pros:
- More expressive long term.
- Can support advanced policies such as "use terse mode for approvals" or "ask before changing dependencies".

Cons:
- Too much surface area for the first version.
- Requires conflict resolution UI and a clearer rule precedence model.
- Increases the chance users confuse behavior policy with security permissions.

Reuses: Settings and runtime prompt composition, but would need more new architecture.

## Recommended Approach

Choose Approach B: Participant Rules MVP.

It gives users the feature they asked for with the smallest clean extension to the current architecture. It preserves the existing mental model:

- Roles are reusable identities.
- Participants bind role, provider, model, avatar, mode, permissions, and now behavior rules.
- Runtime sessions receive a resolved instruction snapshot.

## Data Model

Add a reusable rule type:

```ts
interface ChatBehaviorRuleConfig {
  id: string;
  label: string;
  instructions: string;
  version: number;
  updatedAt: string;
}
```

Attach rule IDs to saved participants and copied chat participants:

```ts
interface ChatParticipantConfig {
  behaviorRuleIds?: string[];
}

interface ChatParticipant {
  behaviorRuleIds?: string[];
}
```

Store resolved snapshots on runtime sessions:

```ts
interface ChatBehaviorRuleSnapshot {
  id: string;
  label: string;
  instructions: string;
  version: number;
}
```

## Runtime Behavior

When a participant runs:

1. `ChatService` resolves the participant's role.
2. `ChatService` resolves attached behavior rule IDs from current settings.
3. It appends a `Participant Behavior Rules` section to the role instructions.
4. It stores the resolved rule snapshots on `ChatParticipantSession`.
5. It treats changed rule content or rule versions as instruction refresh input.
6. It includes role instruction hash and selected rule versions in warm-session cache keys.

If a rule was deleted after a participant referenced it, the settings normalization and participant validation should remove or reject the stale reference instead of producing a broken prompt.

## UI Behavior

Settings gets a new `Rules` section:

- list existing behavior rules
- create a rule
- edit rule name and instructions
- delete a rule

Participant editors get a `Behavior rules` picker:

- checkbox list of available rules
- selected rules are stored as rule IDs on the participant preset
- if no rules exist, show an empty state

The MVP does not need per-chat inline rule editing. Users can create/edit reusable rules in Settings, then attach them to saved participants.

## Open Questions

- Should active chat participants be editable in place later, or should users create a new participant preset and add it to the chat?
- Should there be built-in starter rules, such as "Concise", "Skeptical reviewer", and "Always verify"?
- Should rules eventually be exportable with participant presets?
- Should rule precedence be user-controlled, or should settings order remain the only order?

## Success Criteria

- A user can create a reusable behavior rule in Settings.
- A user can attach one or more behavior rules to a saved participant.
- Starting or adding a participant carries the selected rule IDs into the chat participant record.
- Participant runtime instructions include the selected behavior rule text.
- Editing a behavior rule refreshes future session instructions.
- Deleting a behavior rule removes it from saved participant presets.
- Existing role behavior, permissions, skill mentions, and chat concurrency continue to pass targeted tests.

## Next Steps

1. Implement settings and shared wire types for `ChatBehaviorRuleConfig`.
2. Add Settings UI for behavior rule create/edit/delete.
3. Add rule selection to saved participant editors.
4. Resolve and append behavior rules in `ChatService` session instruction setup.
5. Add regression coverage for behavior-rule instruction merging.
6. Run `make typecheck`, `make build`, and targeted chat tests.

## What I Noticed About This Request

- The wording was "rules for participants behavior", not "new roles", which points to reusable overlays rather than more role presets.
- The existing MVP TODO already separates rule management from participant management, so this feature should become first-class instead of being hidden inside role text.
- The safest first version is narrow: saved participant rules only. Global conditions and per-chat overrides can come after the basic model proves useful.
